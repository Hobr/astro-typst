# Design: Astro 7.1 and Typst 0.15 compatibility

## Scope and task shape

This remains one implementation task. Dependency selection, integration code,
demo compilation, and compatibility verification form one ordered migration:
none is independently releasable or meaningfully verifiable without the
others. A parent/child split would add coordination without creating separate
deliverables.

## Current data flow

```text
Astro config
  -> typst integration (`src/lib/integration.ts`)
  -> registers `.typ` page/content types and the custom renderer
  -> installs the Vite plugin (`src/lib/vite.ts`)
  -> selects HTML or SVG target
  -> typst.ts NodeCompiler (`src/lib/typst.ts`)
  -> optional HAST/TypstX processing or rich SVG post-processing
  -> generated Astro component
  -> demo pages and content collection output
```

The system Typst binary in the Nix shell is a developer convenience. It is not
part of this runtime path and must not be used to claim embedded compatibility.

## Dependency design

### Target development matrix

- Astro: 7.1.1
- Vite: 8.1.x
- TypeScript: supported 6.x line
- typst.ts packages: exactly 0.8.0-rc3 for local development
- Embedded Typst: 0.15.0
- Node: Astro-supported 22.12 or newer; local Nix shell uses 24.18.0

`package.json` will distinguish two concerns:

1. `peerDependencies` continue to express the existing consumer-selected
   compiler policy. The broad typst.ts and Astro ranges are retained because
   the used APIs remain compatible.
2. `devDependencies` provide a complete, reproducible local environment. The
   three typst.ts packages are pinned to 0.8.0-rc3 and Cheerio is installed
   locally, matching packages imported or expected by the repository build.

`astro-integration-kit` will be removed. Its latest release excludes Astro 7,
and the two utilities used here do not justify retaining an unsupported peer:

- `createResolver` exists only to calculate the directory passed to
  `watchDirectory`.
- `watchDirectory` triggers a broad development restart for the package source.
  The actual `.typ` HMR contract is already owned by the Vite plugin and Astro's
  page watcher.

The user's other dependency updates remain in place unless a build proves a
specific incompatibility. The lockfile will be regenerated by pnpm rather than
edited manually.

Integration config normalization will spread the complete caller config before
applying defaults to `options` and `target`. This preserves `emitSvg`,
`emitSvgDir`, `htmlMode`, `fontArgs`, and future additive fields at the
integration-to-Vite boundary.

Astro's static asset rearranger tracks its own SSR asset registry, so arbitrary
assets emitted from this custom transform can disappear before final output.
The Astro integration will collect those SVG sources during transforms and
write them beneath the final `astro:build:done` `dir`; direct Vite consumers keep
the normal `emitFile` fallback. This preserves the configured `emitSvgDir`,
including the demo's existing `.astro/typst` path.

## TypeScript and declaration output

The compiler will remain responsible for emitting the package's JavaScript and
declarations under `dist/`. `tsconfig.json` will declare `rootDir: "src"` so the
existing `dist/index.js` and `dist/index.d.ts` package contract is stable under
TypeScript 6.

Astro 7 rejects the legacy `src/content/config.ts` location and collections
without a loader. The demo will move to `src/content.config.ts`, use the public
`glob()` loader for `.typ` files, and give its `collections` export the narrowest
public annotation needed for portable declarations. Collection routes will use
`entry.id` and `render(entry)`. Declaration emission will not be disabled, and
the demo will not be removed from checking merely to hide an error.

## Astro integration boundary

`src/lib/integration.ts` will keep the existing internal content hooks because
Astro 7.1 still supplies them and the integration needs custom `.typ` page and
content entry support. The local `SetupHookParams` type remains the single
owner of that non-public contract.

The Astro Integration Kit import, resolver, and `watchDirectory` invocation
will be deleted. No replacement helper is introduced. The redundant Rollup
node resolver will also be removed: local debug mode will provide Astro a
`file:` URL anchored to `import.meta.url`, while published mode retains the
package subpath. Renderer registration,
content entry extraction, frontmatter querying, externalization of the native
compiler, and injected `.typ` module types remain unchanged unless the target
build exposes a concrete Astro 7 incompatibility.

Astro 7's stricter Rust template compiler, JSX whitespace default, and Satteri
Markdown default will be validated through the demo. The design does not force
legacy `compressHTML` or Markdown settings preemptively.

## Vite plugin boundary

`src/lib/vite.ts` continues to own `.typ` recognition, target selection, HMR,
asset emission, and generated Astro module code. Its transform result will add
`moduleType: "js"` so Vite 8/Rolldown treats generated code from a `.typ` ID as
JavaScript.

This metadata is additive. Older Vite/Rollup consumers can ignore it, which
supports retaining the existing Astro peer range. The generated module and the
HTML/SVG branches otherwise remain stable until tested output demonstrates a
required change.

Build-mode branching will receive `astro:config:setup`'s command when the Vite
plugin is created. Per-hook `this.environment.mode` and
`configResolved.command` remain direct-plugin fallbacks. It will not use
`import.meta.env.PROD`: the plugin executes in Node's config environment and
Astro can transform content during sync before production Vite environments
run, which would silently inline base64 SVG data even when `emitSvg` requests
build assets.

## Typst compiler contract

The three typst.ts development packages stay on the same RC3 version. A small
Node compatibility probe will:

1. compile `#metadata(sys.version)<version>`;
2. query `<version>` and require `version(0, 15, 0)`;
3. compile a minimal rich SVG;
4. require `typst-doc` and `typst-text` markers.

The probe verifies the actual native package selected on the host. It does not
substitute for the Astro demo build, which validates the public integration
path, HTML export, SVG export, frontmatter, content collections, slots, and
emitted assets.

## Documentation and CI

README badges and installation guidance will add Typst 0.15 and the matching
0.8.0-rc3 package set without deleting the older version-selection examples.

Both publish jobs must run on an Astro 7-supported Node version. The existing
Node 22 job remains valid; the Node 20 job will move to Node 22 or newer. The
normal build command will include the lightweight compiler probe so release
builds cannot silently use Typst 0.14.

## Compatibility and trade-offs

- RC3 is accepted because it is the only published typst.ts line embedding
  Typst 0.15.0. Pinning all three packages avoids mixed native/renderer APIs.
- Broad peer ranges preserve the project's established consumer choice, but
  the guaranteed matrix for this task is Astro 7.1 plus Typst 0.15.
- Removing directory-wide reload behavior is preferable to keeping an
  unsupported integration dependency. Imported `.typ` HMR remains covered by
  the existing Vite watcher and will receive a development smoke test.
- Direct Astro runtime imports remain a maintenance risk, but replacing the
  custom renderer is outside scope. The target build and demo routes provide
  regression coverage for those imports.

## Rollout and rollback

There is no data migration and no deployment in this task. Changes are local to
manifests, integration code, a compatibility probe, docs, and CI.

Rollback points:

1. Dependency/lockfile changes can be reverted together without touching the
   rendering implementation.
2. Astro Integration Kit removal and Vite module metadata are isolated source
   edits and can be reverted independently if target builds disprove the
   design.
3. No package version bump or publish occurs, so a failing migration can be
   corrected before release.
